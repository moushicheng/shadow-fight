# Shadow Fight — 完整游戏开发计划

## 目标

实现策划文档中描述的完整 Roguelike 卡牌构筑游戏，可在抖音小游戏平台运行。验收标准：

1. 玩家能完成一局完整的 10 层爬塔流程
2. ATB 战斗系统可正确运行，所有状态效果（霜蚀/灼烧/毒药/诅咒/汲取）生效
3. 卡组构筑循环可用（获取、移除、升级、调序）
4. 残影 PVP 可用（至少 AI 残影可对战）
5. 竖屏 UI 布局符合设计规范

---

## 整体架构

```
shadow-fight/
├── assets/
│   ├── scripts/
│   │   ├── core/           # 纯逻辑层（不依赖引擎，可单元测试）
│   │   │   ├── battle/     # ATB 战斗引擎
│   │   │   ├── card/       # 卡牌系统
│   │   │   ├── character/  # 角色属性系统
│   │   │   ├── deck/       # 卡组管理
│   │   │   ├── event/      # 事件系统
│   │   │   ├── faction/    # 流派池系统
│   │   │   ├── ghost/      # 残影 PVP
│   │   │   ├── monster/    # 野怪系统
│   │   │   ├── relic/      # 遗物系统
│   │   │   ├── run/        # 单局状态管理（RunState）
│   │   │   ├── shop/       # 商店系统
│   │   │   └── season/     # 赛季系统
│   │   ├── data/           # 静态数据定义（卡牌表、野怪表、遗物表、事件表）
│   │   ├── ui/             # UI 组件脚本
│   │   │   ├── battle/     # 战斗界面
│   │   │   ├── deck/       # 卡组管理界面
│   │   │   ├── shop/       # 商店界面
│   │   │   ├── event/      # 事件界面
│   │   │   ├── selection/  # 路线/残影选择界面
│   │   │   ├── reward/     # 奖励选择界面
│   │   │   └── common/     # 通用 UI 组件
│   │   ├── managers/       # 全局管理器（场景切换、音频、存档）
│   │   └── types/          # TypeScript 类型定义
│   ├── scenes/             # 场景文件
│   ├── prefabs/            # 预制体
│   ├── textures/           # 图片资源
│   ├── fonts/              # 字体
│   └── audio/              # 音效
├── docs/                   # 策划文档（已有）
├── context/                # 知识库
└── requirements/           # 需求追踪
```

**架构原则**：`core/` 层是纯 TypeScript 逻辑，不依赖 Cocos Creator API，可独立测试。`ui/` 层负责将 core 的状态映射到可视化。`data/` 层存放所有静态配置数据。

> **目标**：实现一局完整的 10 层爬塔流程逻辑

| # | 任务 | 产出 | 依赖 |
|---|------|------|------|
| 2.1 ✅ | RunState 管理器 | `core/run/RunManager.ts` — 创建/保存/恢复单局状态 | Phase 0 |
| 2.2 ✅ | 流派池系统 | `core/faction/FactionPool.ts` — 随机抽 2 流派、限制卡牌获取范围 | 2.1 |
| 2.3 ✅ | 层级推进 | `core/run/FloorManager.ts` — 10 层结构、难度曲线、节点流转 | 2.1 |
| 2.4 ✅ | 事件系统框架 | `core/event/EventManager.ts` — 事件抽取（正面/中性/负面概率）、效果执行 | 2.3 |
| 2.5 ✅ | 路线选择系统 | `core/run/RouteSelector.ts` — 精英/普通A/普通B三选一 + 赏金挑战；生命机制（5颗心） | 2.3 |
| 2.6 | 野怪生成系统 | `core/monster/MonsterGenerator.ts` — 按层数生成属性/卡组/遗物 | 2.3 |
| 2.7 | 商店系统 | `core/shop/ShopManager.ts` — 商品生成、购买、服务（移除/升级/调序） | 2.3 |
| 2.8 | 遗物系统 | `core/relic/RelicManager.ts` — 获取、触发时机、效果执行 | Phase 1 |
| 2.9 | 卡牌奖励生成 | `core/card/RewardGenerator.ts` — 战斗后 3 选 1 卡牌、品质概率 | 2.2 |
| 2.10 | 战斗奖励结算 | `core/run/RewardSettlement.ts` — 金币/卡牌/遗物奖励分发 | 2.9, 2.8 |

**关键验证点**：

**2.1 RunState 管理器**
- 创建新局时，四维属性总和 = 40，且 SPD ≥ 3
- 重摇属性：每局仅允许 1 次免费重摇，第二次调用应被拒绝
- 保存后恢复的 RunState 与保存前完全一致（卡组顺序、金币、当前层数、遗物列表等）
- 中途退出再恢复，应回到上次所在的节点位置（如"第 3 层·第 2 循环·商店"）

**2.2 流派池系统**
- 每局从 10 流派中恰好抽取 2 个不同流派，不会出现重复
- 卡牌获取限制：战斗奖励 / 商店 / 事件中生成的卡牌只来自流派池内 + 通用（COMMON），不会出现池外流派卡
- 大量重复生成（如 1000 局），10 个流派的被选中频率应趋于均匀

**2.3 层级推进**
- 每层恰好包含 6 个节点：事件 → 路线三选一 → 事件 → 事件 → 残影三选一 → 商店
- 1-2 层路线节点锁定为普通战斗，无三选一
- 3 层起路线节点提供精英/普通/未知事件三选一
- 第 10 层 Boss 层：无事件、无商店，仅残影战斗（1.5 倍 HP）
- 完整走完 10 层后游戏状态标记为"胜利"

**2.4 事件系统框架**
- 1-3 层事件概率：正面 50% / 中性 30% / 负面 20%
- 4-6 层事件概率：正面 35% / 中性 35% / 负面 30%
- 7-9 层事件概率：正面 20% / 中性 35% / 负面 45%
- 大量抽样（如 10000 次）下各类型事件占比符合上述概率（允许 ±3% 误差）
- 事件效果正确执行：如"属性永久 -1"应修改 RunState 的基础属性且后续战斗反映该变化

**2.5 路线选择系统 + 生命机制**
- 三选一：精英战斗 / 普通战斗 A / 普通战斗 B，每个选项展示怪物名称
- 两个普通战斗选项的怪物不同（从野怪池中分别抽取）
- 精英战斗的野怪强度 = 当前层 + 2 的普通野怪强度（如第 4 层精英 ≈ 第 6 层普通）
- 赏金挑战出现概率约 10%（大量抽样验证），赏金怪物强度 = 精英参数 × 1.5
- 赏金胜利奖励 = 标准奖励 + 额外随机 1 个遗物
- 选择精英路线后奖励金币 ×1.5，且必出稀有及以上品质卡牌
- 1-2 层锁定为普通战斗（无三选一），3 层起解锁
- 生命机制：玩家每局 5 颗心，战斗失败消耗 1 颗心后继续流程，心耗尽则 Game Over

**2.6 野怪生成系统**
- 野怪属性总点数 = 25 + 层数 × 3（如第 5 层 = 40 点）
- 野怪卡组大小 = 3 + 层数（如第 5 层 = 8 张牌）
- 1-3 层野怪无遗物；4-6 层携带 1 个普通遗物；7-9 层携带 1-2 个遗物
- 流派野怪仅在 3 层及以上出现，精英野怪仅在 6 层及以上出现

**2.7 商店系统**
- 商品栏位正确：5 张卡牌 + 1-2 瓶药水 + 0-1 个遗物（50% 概率）+ 2 个服务
- 服务从 3 种（移除/升级/调序）中随机出现 2 种
- 每次进入商店只能使用 1 次服务
- 服务价格递增：首次 50 金，每使用 1 次后所有服务永久 +25 金（第 2 次 75、第 3 次 100…）
- 移除卡牌后卡组张数 -1，且被移除的卡不再出现
- 调整卡序后卡组内容不变，仅目标卡牌位置改变
- 购买的卡牌自动追加到卡组末尾

**2.8 遗物系统**
- 遗物持有无上限
- 遗物品质掉落概率：普通 60% / 稀有 30% / 传说 10%
- 遗物效果在正确时机触发：如"每场战斗开始时获得 5 护甲"应在 BattleInitializer 阶段生效
- 同一遗物不重复获取（或按设计允许叠加时正确叠加）

**2.9 卡牌奖励生成**
- 战斗胜利后生成 3 张候选卡牌供玩家三选一
- 候选卡牌全部来自流派池内 + 通用，不出现池外卡
- 精英路线必出稀有及以上品质卡牌

**2.10 战斗奖励结算**
- 普通战斗给予标准金币奖励
- 精英路线金币奖励 = 标准 × 1.5
- 高星残影金币奖励 +30%，遗物掉落概率 +10%
- 每场战斗胜利有 20% 概率额外掉落遗物（3 选 1）
- 获取的卡牌/遗物正确写入 RunState

## 优先级排序原则

1. **核心循环优先**：先让战斗能跑（Phase 1），再让一局能玩（Phase 2），最后上 UI（Phase 4-5）
2. **纯逻辑先行**：core 层不依赖引擎，可快速迭代和测试
3. **数据可并行**：Phase 3 的内容填充可与 Phase 1-2 同时推进
4. **MVP 思维**：每个 Phase 结束都有可验证的产出
5. **延迟决策**：服务端、社交功能、美术资源可后置，先用占位资源

## 风险与待确认

| 风险 | 影响 | 缓解方案 |
|------|------|----------|
| 卡牌数量巨大（80-120 张） | Phase 3 可能耗时超预期 | 先实现 3-5 个核心流派（冰/火/毒 + 通用），其余分批补充 |
| ATB 战斗的 UI 表现力 | 自动战斗可能观感单调 | 投入足够的动效和反馈设计（Phase 9） |
| 抖音小游戏包体限制 | 资源过大无法发布 | 图片压缩、动态加载、资源分包 |
| 数值平衡工作量大 | 45 种流派组合难以手动测试 | 开发自动化模拟器（10.6）批量验证 |
| 策划中标记"待设计"的内容 | Boss 战、成就系统、完整事件列表等尚未设计 | 按优先级逐步补充，不阻塞主流程 |

## 建议的开发顺序（首次迭代聚焦 MVP）

如果时间有限，建议 **Phase 0 → 1 → 2（精简版）→ 3（核心 3 流派）→ 4 → 5（核心界面）→ 7（联调）** 作为首次可玩版本，其余后续迭代补充。

精简版 MVP 的范围：
- 3 个流派（冰 + 火 + 通用），其余 8 个流派后补
- 5 层而非 10 层
- 10 个遗物（而非 30-50 个）
- 10 个事件（而非 25-30 个）
- 用 AI 残影代替真实 PVP
- 占位美术资源

**预计 MVP 工期：15-20 天**
