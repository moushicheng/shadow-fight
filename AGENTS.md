# shadow-fight — AI 协作规范

## 项目简介

**类型**：Roguelike 卡牌构筑 + 异步 PVP

**核心卖点**：随机属性、流派构筑、PvE + 异步 PvP——每局随机生成四维属性和可选流派池，通过爬塔中的事件、野怪战斗、残影对决、商店不断构筑卡组。每层先挑战野怪获取资源，再对抗其他玩家留下的"残影"数据。

**参考游戏**：杀戮尖塔（Slay the Spire）

**平台**：抖音小游戏（轻量级，单局 5-15 分钟）

## 协作原则

- 先理解问题，再动手写码
- 不确定时说"我不知道"，不编造
- 重要决策先展示推理过程，再给出结论
- 每次修改考虑可维护性
- 优先中文沟通，技术术语不强行翻译

## 知识库使用

- 开始工作前，先检查 `context/` 下是否有相关知识
- 遇到新的经验/踩坑，提议写入 `context/` 对应领域
- 知识文档只放已验证的信息，未确认的标注 `[待确认]`

## 需求工作流

1. 新需求 → 创建 `requirements/{id}/brief.md`，描述做什么和为什么
2. AI 根据 brief 生成 `plan.md`，与用户确认后开始
3. 开发过程中 → 关键节点更新 `progress.txt`（事实+决策+下一步）
4. 踩坑/发现 → 记入 `notes.md`
5. 需求完成 → 有价值的经验迁移到 `context/`

## 会话恢复

新会话开始时，如果用户说"继续"：

1. 读 `progress.txt` 最近的条目，了解上次做到哪
2. 读 `plan.md` 了解整体计划和当前位置
3. 用 2-3 句话告诉用户上次的状态和建议的下一步
4. 等用户确认后再动手

- 不在主分支直接开发（如果项目有分支策略）

## 4. context/ — 知识库

### 设计原则

- **只放已验证的知识**，不放猜测和草稿
- **按领域组织**，不按时间或需求组织
- **每个领域一个 INDEX.md**，方便 AI 快速定位

### 领域划分建议

根据项目类型选择合适的领域划分：

```
context/
├── architecture/        # 架构决策、技术选型、系统边界
│   ├── INDEX.md
│   ├── tech-stack.md          # 技术栈选型和理由
│   └── deployment.md          # 部署架构
├── business/            # 业务规则、领域知识
│   ├── INDEX.md
│   └── {业务概念}.md
├── experience/          # 踩坑经验、最佳实践
│   ├── INDEX.md
│   └── {主题}.md
└── services/            # 外部服务/API 的对接知识
    ├── INDEX.md
    └── {服务名}.md
```

### INDEX.md 格式

每个领域目录下的 INDEX.md 采用卡片式索引：

```markdown
# {领域名} 知识索引

## {文档标题}

- **路径**：`{filename}.md`
- **摘要**：一句话说明文档内容
- **关键词**：用于 AI 检索的关键词
- **更新时间**：YYYY-MM-DD

## {另一个文档标题}

...
```

### 知识生命周期

```
开发中发现新知识
    ↓
先记到 requirements/{id}/notes.md（与需求绑定）
    ↓
需求完成后，有复用价值的 → 迁移到 context/{domain}/
    ↓
context/ 中的知识被后续需求引用 → 持续更新
```

## 5. requirements/ — 需求追踪

### 什么时候创建需求

不是所有工作都需要创建需求目录。判断标准：

| 情况                      | 是否创建 |
| ------------------------- | -------- |
| 预计超过 1 小时的功能开发 | 创建     |
| 可能跨多次会话的工作      | 创建     |
| 涉及重要设计决策的改动    | 创建     |
| 5 分钟的 bugfix           | 不需要   |
| 简单的配置修改            | 不需要   |

### 需求 ID 命名

简短、可读、有辨识度：

```
requirements/
├── rbac/              # 权限控制
├── rate-limit/        # 限流
├── user-export/       # 用户导出
└── fix-login-race/    # 修复登录竞态
```

### 各文件详细说明

#### brief.md — 需求种子

```markdown
# {需求标题}

## 做什么

（1-3 句话描述要实现的功能或要解决的问题）

## 为什么

（背景、动机、业务价值）

## 涉及范围

（影响的模块、服务、文件范围）

## 约束和边界

（性能要求、兼容性、不做什么）
```

**要点**：brief 写完后交给 AI，AI 据此生成 plan.md。brief 一旦确认，通常不再大改。

#### plan.md — 执行计划（AI 生成）

AI 读完 brief.md 后，结合 context/ 中的项目知识，生成具体的执行计划：

```markdown
# {需求标题} — 执行计划

## 目标

（从 brief 提炼的验收标准）

## 步骤

1. （具体的第一步）
2. （具体的第二步）
3. ...

## 风险和待确认

- （已识别的风险点）
- （需要用户确认的信息）

## 相关上下文

- `context/architecture/tech-stack.md` — 技术栈约束
- （其他相关的 context 文档引用）
```

**要点**：plan 是活的，执行过程中可以调整。每个需求的 plan 不同，不套模板。

#### progress.txt — 流水日志

```
[YYYY-MM-DD HH:MM] {事件类型}: {内容}
```

事件类型只有几种：

| 类型     | 含义             | 示例                                                     |
| -------- | ---------------- | -------------------------------------------------------- |
| `开始`   | 启动某项工作     | `开始: 设计权限模型`                                     |
| `完成`   | 完成某项工作     | `完成: RBAC 中间件，支持角色+权限二级`                   |
| `决策`   | 做了一个选择     | `决策: 用 Casbin 而不是自研（原因：成熟度高，社区活跃）` |
| `阻塞`   | 遇到问题暂停     | `阻塞: 等确认权限粒度是页面级还是按钮级`                 |
| `下一步` | 标记接下来要做的 | `下一步: 写迁移脚本，把现有管理员映射到超级管理员角色`   |

**要点**：

- 只追加，不回写修改历史记录
- `决策` 必须带理由——这是最有恢复价值的信息
- `下一步` 是会话恢复的关键锚点

#### notes.md — 踩坑和发现

```markdown
# {需求标题} — 笔记

## {发现/踩坑标题}

- **现象**：发生了什么
- **原因**：为什么
- **解法**：怎么处理的
- **适用场景**：什么时候会再遇到
```

**要点**：需求结束后，有复用价值的条目迁移到 `context/experience/`。

## 6. Subagent 策略

### 什么时候用 Subagent

| 场景                         | 用主对话还是 Subagent |
| ---------------------------- | --------------------- |
| 需要反复讨论确认的           | 主对话                |
| 一次性的检索/分析任务        | Subagent              |
| 代码审查                     | Subagent              |
| 批量文件搜索/分析            | Subagent              |
| 需要独立上下文窗口的复杂任务 | Subagent              |

### Subagent 使用原则

1. **自包含**：交给 Subagent 的任务必须能独立完成，不需要频繁回来问主对话
2. **明确产出**：告诉 Subagent 要返回什么格式的结果
3. **上下文传递**：把必要的文件路径和背景信息传给 Subagent，不要让它自己猜

## 7. 落地步骤

在新项目中启用这套架构，只需要 3 步：

### 第一步：创建骨架

```bash
# 在项目根目录执行
mkdir -p context/{architecture,business,experience}
mkdir -p requirements
```

### 第二步：编写 AGENTS.md

复制第 3 节的模板到项目根目录，填入项目具体信息。

### 第三步：初始化知识库

把项目已有的关键知识写入 context/：

```bash
# 至少写这几个文件
touch context/architecture/INDEX.md
touch context/architecture/tech-stack.md  # 技术栈和选型理由
touch context/business/INDEX.md
touch context/experience/INDEX.md
```

然后就可以开始工作了。遇到新需求时创建 requirements/{id}/，日常开发中持续丰富 context/。

## 8. 反模式清单

以下是应该避免的做法：

| 反模式                          | 为什么不好                            | 应该怎么做                          |
| ------------------------------- | ------------------------------------- | ----------------------------------- |
| 所有工作都创建需求目录          | 5 分钟的活也走流程，overhead 大于价值 | 只有跨会话/有决策的工作才创建       |
| progress.txt 记录 AI 的思考过程 | 膨胀快、恢复时噪音多                  | 只记事实、决策、下一步              |
| context/ 放未验证的信息         | 错误知识比没有知识更危险              | 未确认的放 notes.md，确认后才迁移   |
| brief.md 写成长篇需求文档       | 种子应该精炼，AI 会展开               | 控制在 10-20 行以内                 |
| plan.md 写死不改                | 计划赶不上变化                        | plan 是活的，边做边调整             |
| notes.md 堆积不迁移             | 知识困在需求目录里无法复用            | 需求完成后，有价值的迁移到 context/ |

## 9. 快速参考卡

```
新需求：
  1. mkdir requirements/{id}
  2. 写 brief.md（做什么+为什么）
  3. 让 AI 生成 plan.md
  4. 开始干活，关键节点更新 progress.txt

恢复会话：
  1. 告诉 AI "继续 {id}"
  2. AI 读 progress.txt + plan.md → 告诉你上次状态
  3. 确认后继续

完成需求：
  1. 最后一条 progress.txt 标记完成
  2. notes.md 里有价值的 → 迁移到 context/
  3. requirements/{id}/ 保留作为历史记录（或归档）

日常沉淀：
  遇到新知识/踩坑 → context/{domain}/{topic}.md
  更新对应的 INDEX.md
```
